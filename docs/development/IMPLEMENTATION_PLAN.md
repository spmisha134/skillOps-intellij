# Implementation Plan: BuildSkill Intellij for IntelliJ

## Summary

Build the first version of BuildSkill Intellij as an IntelliJ IDEA plugin that creates and validates repository-scoped Codex/OpenAI skills under:

```text
.agents/skills/<skill-name>/
```

The implementation replaces the IntelliJ template package with:

```text
com.spmisha134.buildskill
```

Keep IntelliJ-dependent integration separate from pure Kotlin generation and validation logic. The repository follows the sibling `plugin-reference/sonarlint-intellij` structure in a reduced form by package organization, while remaining a single Gradle module.

## Code structure

```text
src/main/kotlin/com/spmisha134/buildskill/
  actions/          # IntelliJ Action System only
  ui/               # IntelliJ/Swing dialog and form code
  presentation/     # IntelliJ messages and validation result presentation

  model/            # Pure Kotlin data classes
  generator/        # Pure Kotlin normalization, rendering, pathing, generation
  validator/        # Pure Kotlin validation orchestration
    rules/          # Pure Kotlin validation rules

src/main/resources/templates/
  generated-skill/  # Templates used by the domain renderers

src/test/kotlin/com/spmisha134/buildskill/
  generator/        # Pure domain unit tests
  validator/        # Pure validation unit tests
```

Pure Kotlin packages must not import IntelliJ APIs. IntelliJ actions and dialogs convert IDE state into plain request objects, then delegate to generator and validator components.

## Implementation changes

- Configure plugin identity:
  - plugin name: `BuildSkill Intellij`
  - plugin id: `com.spmisha134.buildskill`
  - package root: `com.spmisha134.buildskill`
- Remove template tool window, startup activity, sample service, sample tests, and related registrations.
- Add `BuildSkill` under the project-view `New` menu and add `Validate BuildSkill` to validation entry points.
- Add a creation dialog with required fields:
  - skill name
  - description
- Add optional generation checkboxes, defaulted to enabled:
  - `scripts/`
  - `assets/`
- Always generate `agents/openai.yaml`.

## Generation rules

- Implement `SkillDefinition`, `SkillGenerationRequest`, and `SkillGenerationResult`.
- Implement `SkillNameNormalizer`, `SkillPathResolver`, `SkillTemplateRenderer`, `ReferenceTemplateRenderer`, and `SkillGenerator`.
- Always generate skills under:

```text
<project-root>/.agents/skills/<normalized-name>/
```

- Even when the user right-clicks a subfolder, use the project root as the skill base directory.
- Reject duplicate skill folders without overwriting.
- Always generate `SKILL.md`.
- Always generate:

```text
references/instructions.md
references/validation.md
references/examples.md
```

- Optionally generate:

```text
scripts/
assets/
```

- Always generate `agents/openai.yaml`.

- Generated `SKILL.md` must remain small and delegate detailed instructions to `references/`.
- Generated `SKILL.md` must require the agent to search for `AGENTS.md`/`agents.md`, `CLAUDE.md`/`claude.md`, and `GEMINI.md`/`gemini.md` before suggesting code structure, package layout, testing, naming, implementation, or architecture changes.

## Validation rules

Validation is rule-based and deterministic.

Initial rules:

- skill is under `.agents/skills/`
- `SKILL.md` exists
- `SKILL.md` starts with YAML front matter
- front matter includes `name`
- front matter includes non-blank `description`
- `SKILL.md` is small, warning above `1200` characters
- required reference files exist
- existing reference files are not empty
- project instruction discovery is present in `SKILL.md`
- skill folder name is normalized and safe
- `agents/openai.yaml` exists and is not obviously malformed

## Test plan

Pure unit tests:

- `SkillNameNormalizerTest`
- `SkillPathResolverTest`
- `SkillTemplateRendererTest`
- `ReferenceTemplateRendererTest`
- `SkillGeneratorTest`
- `SkillValidatorTest`
- rule-specific validator tests

Key scenarios:

- right-clicked subfolder still resolves generation target to project root
- duplicate skill folders are rejected
- required files are always generated
- optional `scripts/` and `assets/` folders are generated by default and skipped only when disabled
- `SKILL.md` remains small
- `SKILL.md` contains project instruction discovery for all required agent files
- missing `SKILL.md` fails validation
- missing references fail validation
- empty references warn
- malformed `agents/openai.yaml` warns

## Verification

Run:

```bash
./gradlew check
./gradlew buildPlugin
./gradlew runIde
```

Manual sandbox verification:

1. Open a test project.
2. Right-click the project root.
3. Select `New → BuildSkill`.
4. Fill required fields.
5. Create the skill.
6. Confirm files exist under `.agents/skills/<skill-name>/`.
7. Confirm `SKILL.md` is small.
8. Confirm reference files are not empty.
9. Right-click the skill folder.
10. Select `Validate BuildSkill`.
11. Confirm valid skill passes.
12. Delete `references/examples.md`.
13. Validate again.
14. Confirm validation fails with a useful message.
15. Try creating a duplicate skill.
16. Confirm duplicate creation is rejected.
